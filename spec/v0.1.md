## 1. Introduction

### 1.1 Problem

AI agents order services, sign up for platforms, manage tasks, and call external APIs. But the entire onboarding infrastructure assumes a human on the other end.

When an agent tries to register with a service today, it runs into three fundamental problems:

1. **Technical:** forms, CAPTCHAs, and UIs are designed for humans — agents have to emulate them, which is fragile and often violates terms of service
2. **Identity:** the service has no idea who is behind the agent, what the agent is authorized to do, or who to contact when something goes wrong
3. **Legal:** there is no mechanism to confirm that the agent is acting within delegated authority, or who bears responsibility for its actions

### 1.2 Solution

DAP (Delegated Agent Protocol) is an open protocol that defines a standard way for agents and services to interact during onboarding. The protocol addresses three problems:

1. The agent presents a **verifiable credential** — a structured proof of its identity and authority
2. The service **verifies the credential** and accepts the agent as a participant with a clear accountability model
3. A **delegation chain** ties the agent's actions back to a principal — the person or organization responsible for it

### 1.3 Principles

- **Compatibility** — the protocol builds on existing standards (OAuth 2.1, W3C VC) and does not require replacing infrastructure
- **Minimality** — v0.1 covers onboarding only; payments, document signing, and agent-to-agent communication are out of scope
- **Openness** — the spec and reference implementations are released under Apache 2.0
- **Data minimization** — the service receives only the minimum data needed for verification, nothing more

## 2. Core Concepts

### 2.1 Participants

| Term | Definition |
|------|------------|
| **Principal** | A person or organization on whose behalf the agent acts. Bears legal and operational responsibility for all of the agent's actions. |
| **Agent** | A software agent acting autonomously. It is not a legal entity — it operates strictly within the authority delegated by the principal. |
| **Service** | An external platform or application where the agent goes through onboarding. Accepts or rejects the agent based on its credential. |
| **Issuer** | The party that created and signed the agent's credential. Can be: the principal itself (self-issued) or a trusted identity provider (delegated issuer). |
| **DAP Registry** | An optional centralized or federated registry that verifies principals and issues agent credentials. Planned for v0.2. |

### 2.2 Credential Issuance Models

The protocol supports two credential issuance models:

#### Self-issued (principal = issuer)

The principal generates its own keys, signs the credential, and publishes its public key. Typical for organizations that control their own domain.

```
Principal (ООО Ромашка)
  │
  ├─ generates key pair
  ├─ publishes public key at did:web:romashka.ru
  ├─ signs Agent Credential with its private key
  │
  └─► Agent presents credential to service
           └─► Service verifies signature against principal's key
```

#### Delegated issuer (credential issued by a third party)

The principal authenticates with a trusted issuer (OAuth provider, DAP Wallet, KYC service), which issues and signs the credential on the principal's behalf. Typical for individuals.

```
Principal (Vasily Petrov)
  │
  ├─ authenticates with issuer (Google OAuth, DAP Wallet)
  │
  └─► Issuer verifies the principal's identity
           ├─ issues Agent Credential
           ├─ sets issuer_id = its own identifier
           ├─ signs credential with its key
           │
           └─► Agent presents credential to service
                    └─► Service verifies signature against issuer's key
```

### 2.3 Delegation Chain

The central concept of the protocol is an explicit, verifiable chain from principal to agent to specific action:

```
Principal (ООО Ромашка, tax ID 1234567890)
  │
  ├─ issues Agent Credential for agent agent-id-abc123
  │    └─ scope: [register, read_data]
  │    └─ constraints: { expires: '2026-12-31' }
  │
  └─► Agent presents credential to service
           └─► Service verifies signature (principal's or issuer's)
                    └─► Accepts agent, creates account
                         └─► Audit trail tied to principal
```

### 2.4 Accountability Model

DAP deliberately does not create new legal entities. Instead, it formalizes delegation:

- The agent acts as a **tool of the principal** — analogous to a power of attorney that lets a representative act on behalf of a legal entity
- The service accepts the agent **knowing who the principal is** and what the boundaries of the agent's authority are
- In case of an incident, the service contacts **the principal** — not the agent
- The principal can **revoke the credential** at any time — this immediately terminates the agent's authority

> **Note:** DAP does not resolve the question of legal liability for agent actions — that remains governed by applicable law. The protocol provides technical traceability: who granted authority, what authority was granted, when, and what the agent did with it.

## 3. Agent Credential

### 3.1 Structure

An Agent Credential is a signed JSON document that the agent presents during onboarding. The protocol supports two formats:

- **JWT** — maximum compatibility with existing infrastructure
- **W3C Verifiable Credential** — cleaner architecture, DID support

Both formats carry the same information. The service declares supported formats in its discovery document.

#### Required Fields

| Field | Type | Description |
|-------|------|-------------|
| `spec_version` | string | Protocol version. Current: `"dap/0.1"` |
| `agent_id` | URI | Unique agent identifier. Example: `urn:dap:agent:abc123` |
| `principal_id` | URI | Principal identifier. See [3.2 Principal Identifiers](#32-principal-identifiers) |
| `principal_name` | string | Human-readable principal name |
| `principal_type` | enum | `"individual"` or `"organization"` |
| `scope` | string[] | Array of permitted actions. Example: `["register", "read", "write"]` |
| `issued_at` | Unix timestamp | Time the credential was issued |
| `expires_at` | Unix timestamp or null | Expiration time. If `expires_at` is `null` or absent, the credential has no expiry. This is NOT RECOMMENDED for production use. |

> **Signing:** For JWT credentials, the credential is signed as a standard JWT — the signature is the third segment of the dot-separated token. For W3C VC credentials, signing uses the `proof` object per the W3C specification. See format-specific examples below.

#### Optional Fields

| Field | Type | Description |
|-------|------|-------------|
| `issuer_id` | URI | Issuer identifier. **Required** when the credential is signed by a third party (delegated issuer), not by the principal itself. If absent, the credential is treated as self-issued and the signature is verified against the `principal_id` key. |
| `verification` | object | Principal verification level information. See [4. Principal Verification](#4-principal-verification) |
| `constraints` | object | Additional constraints: rate limits, allowed domains, IP ranges |
| `purpose` | string | Free-form description of the agent's purpose |
| `contact_email` | string | Principal's contact email for incident response |
| `revocation_endpoint` | URL | Endpoint that the service queries (GET) to check whether the credential has been revoked globally. See [5.5 Revocation](#55-step-4-credential-revocation) for the expected request/response format. |
| `agent_version` | string | Agent software version |
| `capabilities` | string[] | Machine-readable description of agent capabilities |

> **Scope values** are service-defined free-form strings. There is no standard vocabulary. Services SHOULD document their supported scopes in their API documentation.

### 3.2 Principal Identifiers

The `principal_id` field contains a URI that identifies the principal. The protocol supports the following schemes:

| Scheme | Format | Use Case | Example |
|--------|--------|----------|---------|
| `did:web` | DID | Organizations with a domain | `did:web:romashka.ru` |
| `did:key` | DID | Self-identification by key | `did:key:z6MkhaXg...` |
| HTTPS URI | URL | Principals via OAuth provider | `https://accounts.google.com/user/12345` |
| `mailto:` | URI | Minimal email-based identification | `mailto:vasya@gmail.com` |
| `tel:` | URI | Phone number identification | `tel:+79001234567` |

For **organizations**, `did:web` is recommended — it ties identity to a controlled domain.

For **individuals**, an HTTPS URI from an OAuth provider or `mailto:` is recommended — it provides a verifiable point of contact without requiring domain ownership.

> **Note:** For `mailto:` and `tel:` principal IDs, self-issued credentials are not supported because there is no standard mechanism to resolve a public key. These schemes require a delegated issuer (`issuer_id` must be present). For `did:key`, the public key is embedded in the identifier itself per the did:key specification.

The service **MAY** restrict accepted `principal_id` schemes via the discovery document (`requirements.accepted_principal_schemes` field).

```json
  "scope": ["register", "read_data", "submit_forms"],
  "issued_at": 1741824000,

  "contact_email": "tech@romashka.ru",
  "verification": {
    "level": "domain",

  "constraints": {
    "allowed_domains": ["supplier-portal.com", "b2b-market.ru"],
    "max_registrations_per_day": 10
  },

}
```

## 4. Principal Verification

### 4.1 Verification Levels

The protocol defines standard verification levels for principals. Levels are ordered from lowest to highest assurance. A service specifying `min_verification_level` will accept credentials at that level or any higher level in the ordering. The ordering is: self < email < phone < oauth < domain < document < organization. These are assurance tiers, not cumulative checks — for example, domain verification does not imply that phone verification was also performed.

| Level | What is confirmed | Typical method | Use case |
|-------|-------------------|----------------|----------|
| `self` | Nothing — credential is self-signed | Self-issued key | Development, testing, internal systems |
| `email` | Principal's email address | Confirmation link | Minimal verification, non-critical services |
| `phone` | Phone number | SMS code | Delivery services, messaging platforms |
| `oauth` | Account with an identity provider | Google, Apple, GitHub OAuth | Most consumer services |
| `domain` | Control over a domain | did:web, DNS record | Organizations, B2B |
| `document` | Government-issued identity document | Passport, ID via KYC provider | Financial services, regulated industries |
| `organization` | Legal entity verified | Registry extract, tax ID | B2B, fintech, government services |

### 4.2 The `verification` Field in the Credential

```json
{
  "verification": {
    "level": "oauth",
    "method": "google_oauth2",
    "verified_at": 1741824000,
    "verified_by": "https://dap-wallet.example"
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `level` | string | Yes | One of the standard verification levels |
| `method` | string | Yes | Verification method. Examples: `"did:web"`, `"google_oauth2"`, `"sumsub_kyc"`, `"egrul"` |
| `verified_at` | Unix timestamp | No | Time when verification was performed |
| `verified_by` | URI | No | Identifier of the party that performed verification. Can be omitted if it matches `issuer_id` |

If the `verification` field is absent, the credential is treated as level `self`.

### 4.3 Signature Verification Logic

The service determines whose key to use for signature verification by the following rule:

1. If `issuer_id` is **absent** — the credential is self-issued. The signature is verified against the public key of `principal_id`.
2. If `issuer_id` is **present** — the credential was issued by a third party. The signature is verified against the public key of `issuer_id`.

In both cases, the service must be able to **resolve the public key** from the URI:

- For `did:web:example.com` — fetch `https://example.com/.well-known/did.json`
- For an HTTPS issuer URI — fetch `<issuer_id>/.well-known/dap/jwks.json`

### 4.4 Service Trust Policy

The service defines its own trust policy:

- Which **issuers** to trust (`trusted_issuers` list in the service configuration)
- What **minimum verification level** to require (`min_verification_level` in discovery)
- Which **`principal_id` schemes** to accept (`accepted_principal_schemes` in discovery)

The service is **NOT REQUIRED** to trust any given issuer. Trust federation works like CAs in TLS: the service decides which CAs to trust.

### 4.5 Discovery: Verification Extensions

```json
{
  "dap_version": "0.1",
  "dap_endpoint": "https://example-service.com/dap/v1",
  "supported_formats": ["jwt", "w3c-vc"],
  "supported_signature_algorithms": ["EdDSA", "RS256"],
  "requirements": {
    "principal_types": ["organization", "individual"],
    "min_scope": ["register"],
    "requires_contact_email": true,
    "requires_revocation_endpoint": false,
    "min_verification_level": "oauth",
    "accepted_principal_schemes": ["did:web", "https", "mailto"],
    "trusted_issuers": [
      "https://dap-wallet.example",
      "https://another-issuer.example"
    ]
  }
}
```

Additional fields in `requirements`:

| Field | Type | Description |
|-------|------|-------------|
| `min_verification_level` | string | Minimum verification level. Defaults to `"self"` |
| `accepted_principal_schemes` | string[] | Accepted `principal_id` schemes. If absent, any scheme is accepted |
| `trusted_issuers` | string[] | List of trusted issuers for delegated credentials. If absent, the service accepts any issuer whose signature is valid |
| `requires_contact_email` | boolean | Whether the credential must include a `contact_email` field. Defaults to `false` |
| `requires_revocation_endpoint` | boolean | Whether the credential must include a `revocation_endpoint` field. Defaults to `false` |

## 5. Onboarding Protocol

### 5.1 Overview

Agent onboarding consists of four steps. All steps are performed programmatically — no human involvement, no UI interaction.

```
Agent                              Service
  │                                  │
  │  GET /.well-known/dap            │
  │ ─────────────────────────────►   │  Step 1: Discovery
  │ ◄─────────────────────────────   │
  │  { dap_endpoint, requirements }  │
  │                                  │
  │  POST /dap/v1/register           │
  │  { credential, agent_card }      │
  │ ─────────────────────────────►   │  Step 2: Registration
  │                                  │  (credential verification)
  │ ◄─────────────────────────────   │
  │  { status, access_token,         │
  │    session_id, granted_scope }   │
  │                                  │
  │  [using the service]             │  Step 3: Operation
  │  Authorization: Bearer <token>   │
  │ ─────────────────────────────►   │
  │                                  │
  │  POST /dap/v1/revoke             │  Step 4: Revocation (when needed)
  │ ─────────────────────────────►   │
```

### 5.2 Step 1: Discovery

A DAP-supporting service publishes its metadata at a well-known path:

```
GET https://example-service.com/.well-known/dap
```

Response:

```json
{
  "dap_version": "0.1",
  "dap_endpoint": "https://example-service.com/dap/v1",
  "supported_formats": ["jwt", "w3c-vc"],
  "supported_signature_algorithms": ["EdDSA", "RS256"],
  "requirements": {
    "principal_types": ["organization", "individual"],
    "min_scope": ["register"],
    "requires_contact_email": true,
    "requires_revocation_endpoint": false,
    "min_verification_level": "oauth",
    "accepted_principal_schemes": ["did:web", "https", "mailto"],
    "trusted_issuers": [
      "https://dap-wallet.example"
    ]
  }
}
```

The `min_scope` array lists scopes that the credential MUST include. If the credential's `scope` does not contain all values in `min_scope`, the service MUST reject the registration.

### 5.3 Step 2: Registration

#### Request

```http
POST /dap/v1/register
Content-Type: application/json

{
  "credential": "<JWT or W3C VC>",
  "agent_card": {
    "name": "Procurement Agent v2",
    "description": "Supplier sourcing agent for ООО Ромашка",
    "version": "2.1.0",
    "capabilities": ["search", "compare", "request_quote"],
    "callback_url": "https://romashka.ru/agents/abc123/callback"
  }
}
```

#### `agent_card` Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | Yes | Human-readable agent name |
| `description` | string | No | Free-form description of the agent's purpose |
| `version` | string | No | Agent software version |
| `capabilities` | string[] | No | Machine-readable list of agent capabilities |
| `callback_url` | URL | No | URL for the service to send notifications or callbacks to the agent |

#### Service-Side Verification

The service performs the following checks in order:

1. Parse the credential — format matches JWT or W3C VC
2. Check `spec_version` — the service supports this protocol version
3. Check `expires_at` — the credential has not expired
4. **Determine signing key** — if `issuer_id` is present, resolve the key from `issuer_id`; otherwise, resolve from `principal_id`
5. Verify signature — cryptographically valid for the resolved key
6. **Check issuer** — if `issuer_id` is present, the issuer must be in the service's `trusted_issuers` list (if one is configured)
7. **Check verification level** — `verification.level` >= `min_verification_level` from discovery
8. Check `scope` — the requested scope is within the credential's scope
9. Check revocation — if `revocation_endpoint` is specified, query it
10. Check `constraints` — the agent satisfies all constraints
11. Service business rules — optional checks at the service's discretion

#### Success Response

```http
HTTP 201 Created

{
  "status": "registered",
  "session_id": "ses_9x2km3p",
  "access_token": "eyJhbGciOiJSUzI1NiJ9...",
  "token_type": "Bearer",
  "expires_in": 86400,
  "granted_scope": ["register", "read_data", "submit_forms"],
  "account_id": "acc_romashka_7f3a2b9c",
  "principal_display": "ООО Ромашка"
}
```

#### Success Response Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `status` | string | Yes | Registration status. Value: `"registered"` |
| `session_id` | string | Yes | Unique session identifier for this registration |
| `access_token` | string | Yes | Bearer token for subsequent API requests |
| `token_type` | string | Yes | Token type. Value: `"Bearer"` |
| `expires_in` | integer | Yes | Token lifetime in seconds |
| `granted_scope` | string[] | Yes | Scopes actually granted by the service (may be a subset of requested scope) |
| `account_id` | string | No | Service-side account identifier |
| `principal_display` | string | No | Human-readable display name for the principal as stored by the service |

#### Error Codes

```http
HTTP 4XX

{
  "error": "<error_code>",
  "error_description": "<human-readable description>",
  "error_detail": {}
}
```

| Code | HTTP Status | Description |
|------|-------------|-------------|
| `credential_expired` | 401 | Credential has expired |
| `invalid_signature` | 401 | Signature verification failed |
| `insufficient_scope` | 403 | Requested scope is not covered by the credential's scope |
| `credential_revoked` | 401 | Credential has been revoked |
| `unsupported_version` | 400 | Protocol version is not supported |
| `principal_blocked` | 403 | Principal is blocked by the service |
| `invalid_credential_format` | 400 | Credential does not match the expected format |
| `constraints_violated` | 403 | Constraints violated |
| `untrusted_issuer` | 403 | Issuer (`issuer_id`) is not in the service's trusted issuers list |
| `insufficient_verification` | 403 | Principal's verification level is below the service's minimum requirement |
| `unsupported_principal_scheme` | 400 | The `principal_id` scheme is not supported by the service |
| `rate_limit_exceeded` | 429 | Too many registration attempts; try again later |

### 5.4 Step 3: Authentication on Subsequent Requests

```http
GET /api/v1/suppliers
Authorization: Bearer eyJhbGciOiJSUzI1NiJ9...
X-DAP-Agent-ID: urn:dap:agent:7f3a2b9c
X-DAP-Session-ID: ses_9x2km3p
```

The `X-DAP-Agent-ID` and `X-DAP-Session-ID` headers SHOULD be included for audit trail purposes.

### 5.5 Step 4: Credential Revocation

There are two distinct revocation mechanisms:

**Service-side deregistration.** The principal sends a request to the service to deregister their agent from that specific service. This terminates the agent's session and access with that service.

```http
POST /dap/v1/revoke
Authorization: Bearer <principal_token>

{
  "agent_id": "urn:dap:agent:7f3a2b9c",
  "reason": "agent_decommissioned"
}
```

**Global credential revocation check.** Separately, the `revocation_endpoint` field in the credential is a principal-side endpoint that the service queries (GET) to check whether the credential has been revoked globally — across all services.

```
GET <revocation_endpoint>?agent_id=<agent_id>
```

Expected response:

```json
{ "revoked": false }
```

or

```json
{ "revoked": true, "reason": "key_compromised" }
```

## 6. Implementation Requirements

### 6.1 For Services (Accepting Side)

A service claiming DAP v0.1 support **MUST**:

- Publish a discovery document at `/.well-known/dap`
- Accept registration via `POST /dap/v1/register`
- Support at least one format: JWT or W3C VC
- Support signature algorithms: EdDSA (Ed25519) or RS256
- Correctly determine the signing key based on presence or absence of `issuer_id`
- Perform all required verification checks (section 5.3)
- Return standard error codes
- Maintain an audit log: which agent, which principal, which issuer, when registration occurred, what actions were taken

A service **MAY**:

- Add additional business checks on top of the standard ones
- Request additional fields via discovery `requirements`
- Maintain its own list of trusted issuers (`trusted_issuers`)
- Set a minimum verification level (`min_verification_level`)
- Restrict accepted `principal_id` schemes (`accepted_principal_schemes`)

### 6.2 For Agents (Initiating Side)

An agent implementing DAP v0.1 **MUST**:

- Perform discovery before registration
- Check `requirements` from discovery — in particular `min_verification_level` and `accepted_principal_schemes` — before sending the credential
- Present a correctly signed credential
- Handle all standard error codes, including `untrusted_issuer`, `insufficient_verification`, and `unsupported_principal_scheme`
- Not request scope broader than what is needed for the task (least privilege)

An agent **SHOULD**:

- Include the `X-DAP-Agent-ID` header on requests after registration

### 6.3 For Issuers (Delegated Issuer)

An issuer that issues credentials on behalf of principals **MUST**:

- Publish its public key at `<issuer_id>/.well-known/dap/jwks.json`
- Verify the principal's identity to the level claimed in the `verification.level` field
- Sign the credential with its own key and include its identifier in `issuer_id`
- Not issue credentials with a verification level higher than what was actually performed

An issuer **MAY**:

- Support multiple verification methods (OAuth, KYC, documents)
- Provide a revocation endpoint for issued credentials
- Act as a DAP Wallet — an application that manages credentials on behalf of principals

### 6.4 Security

- All requests **MUST** be made over HTTPS
- Access tokens **MUST** have a limited lifetime (recommended: no more than 24 hours)
- The principal's private key **MUST NEVER** be transmitted to the agent — the agent receives only the signed credential
- The issuer's private key is used **ONLY** on the issuer's side to sign credentials
- The service **MUST** log all registration attempts, including failures
- Rate limiting on the `/dap/v1/register` endpoint is required
- The service **MUST** check `issuer_id` against its trusted issuers list, if one is configured

## 7. Usage Scenarios

### 7.1 SaaS B2B: Organization's Agent Registers with a Work Tool

A company deploys an agent to automate CRM workflows.

1. IT department creates an Agent Credential (self-issued): `principal = ООО Ромашка`, `scope = [register, read_contacts, create_tasks]`, `verification.level = domain`
2. Agent performs discovery on `crm.example.com`
3. Agent sends the credential to `/dap/v1/register`
4. CRM verifies the signature against the key at `did:web:romashka.ru`, creates an account named `"ООО Ромашка / Procurement Agent"`
5. CRM records in audit log: principal, agent_id, date, granted_scope
6. If an incident occurs, CRM emails the principal's `contact_email` — no guessing about who was responsible

### 7.2 Fintech: Agent Opens an Account with a B2B Service

1. A legal entity creates a credential with `principal_type = organization`, legal department's `contact_email`, `verification.level = organization`
2. The service specifies in `requirements`: `requires_contact_email = true`, `principal_types = ["organization"]`, `min_verification_level = organization`
3. During registration, the service sees: the principal is a legal entity, verification level is `organization`, the credential signature is valid
4. The service creates an account tied to the legal entity with restricted scope
5. KYC remains at the principal level — the service can request it separately through standard procedures

### 7.3 Consumer Service: Individual's Agent Books a Restaurant

1. Vasily Petrov opens the DAP Wallet and signs in with Google
2. DAP Wallet verifies him via Google OAuth and creates a credential: `principal_id = https://accounts.google.com/user/12345`, `issuer_id = https://dap-wallet.example`, `verification.level = oauth`
3. Vasily's agent performs discovery on `restaurant-booking.example`
4. The service's discovery specifies: `min_verification_level = oauth`, `trusted_issuers = ["https://dap-wallet.example"]`
5. The agent sends the credential — the service verifies the signature against the issuer's key (`dap-wallet.example`)
6. The service creates an account: `"Vasily Petrov / Booking Agent"`, scope = `["register", "search", "book"]`
7. If an incident occurs (no-show, cancellation) — the service contacts the principal's `contact_email`

### 7.4 Courier Service: Agent Orders a Delivery

1. Anna confirms her phone number in DAP Wallet via SMS code
2. DAP Wallet issues a credential: `principal_id = tel:+79001234567`, `verification.level = phone`
3. The delivery service requires `min_verification_level = phone` — the credential passes verification
4. The agent creates a delivery order on Anna's behalf
